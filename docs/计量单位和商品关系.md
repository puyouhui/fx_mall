## 计量单位与商品绑定设计说明

### 1. 核心表结构与关系

- **`uom_categories`（单位类别表）**
  - `id`: 单位类别 ID（例如「1件/10提/100包」）
  - `name`: 类别名称（展示用）
  - `base_unit_id`: 该类别的**基准单位 ID**（指向 `uom_units.id`）
  - 与前端的对应：
    - 小程序/接口里映射为：`uom_category_id`
    - 后端接口会根据 `base_unit_id` 生成：`uom_base_unit_id`

- **`uom_units`（单位表）**
  - `id`: 单位 ID（例如「件」「提」「包」）
  - `category_id`: 所属单位类别 ID（外键 → `uom_categories.id`）
  - `name`: 单位名称（如「件」「提」「包」）
  - `ratio`: 和基准单位的换算比例
    - 基准单位：`ratio = 1`
    - 比基准大的单位：`1 该单位 = ratio 个基准单位`（`ratio > 1`）
    - 比基准小的单位：`ratio < 1`（通过 “多少个小单位 = 1 基准单位” 反推）
  - `is_base`: 是否为该类别的**基准单位**（`1` 表示是）

- **`products`（商品表）**
  - `uom_category_id`: 绑定的单位类别 ID（外键 → `uom_categories.id`）
  - `specs`: JSON 数组，内部每个规格可以绑定一个**具体单位**：
    - `uom_unit_id`（或前端 camelCase `uomUnitId`）
    - `wholesale_price` / `retail_price` / `cost` / `delivery_count` 等

### 2. 基准单位的维护逻辑

#### 2.1 后端（Go）如何维护 `base_unit_id`

文件：`go_backend/internal/model/uom.go`

- **创建单位 `CreateUomUnit`**
  - 在写入 `uom_units` 后，会调用：
    - `updateCategoryBaseUnitFromUnits(u.CategoryID)`
  - 该函数会：
    - 在 `uom_units` 中查询当前类别下 `is_base = 1` 的记录
    - 如果存在：将其 `id` 写入 `uom_categories.base_unit_id`
    - 如果不存在：将 `base_unit_id` 置为 `NULL`

- **更新单位 `UpdateUomUnit`**
  - 更新 `uom_units` 后，同样调用：
    - `updateCategoryBaseUnitFromUnits(u.CategoryID)`
  - 场景示例：
    - 把某个单位改为基准单位（`is_base = 1`）
    - 取消基准单位（`is_base = 0`）
  - 均会自动刷新该类别的 `base_unit_id`

- **删除单位 `DeleteUomUnit`**
  - 先查出该单位的 `category_id`
  - 删除后调用：
    - `updateCategoryBaseUnitFromUnits(categoryID)`
  - 如果删掉的是当前基准单位：
    - 则类别的 `base_unit_id` 会被重置为 `NULL`

> 总结：**不要直接手改 `uom_categories.base_unit_id`，统一通过单位的基准标记（`is_base`）来驱动，Go 代码会自动同步。**

### 3. 商品与计量单位的绑定规则

#### 3.1 商品层面

- `products.uom_category_id` 必须绑定一个单位类别：
  - 在创建/更新商品时，后端会校验 `uom_category_id` 不能为空。
  - 若为空，将无法通过 API 级验证。
- 一个商品只能绑定 **一个单位类别**，但可以有多条不同单位的规格。

#### 3.2 规格（`specs` JSON）层面

单个规格结构示例（以 JSON 为例）：

```json
{
  "name": "1件装（100包）",
  "wholesale_price": 52,
  "retail_price": 58,
  "cost": 45,
  "description": "≈0.52元/包",
  "delivery_count": 1,
  "uom_unit_id": 4
}
```

约束：

- `uom_unit_id`（如果配置）必须属于该商品的 `uom_category_id` 对应的单位集合。
- 不同规格可以使用 **同一类别下的不同单位**：
  - 例如：同一个商品下同时有「包」「件」「3件装」
- 小程序前端在展示列表价时，会根据 `uom_base_unit_id` 优先选择**基准单位的规格**作为展示价。

### 4. 前端使用方式（小程序）

#### 4.1 接口返回字段

以精选商品为例（`GetSpecialProducts`）：

- 商品字段中会带：
  - `uom_category_id`
  - `uom_base_unit_id`（根据 `uom_categories.base_unit_id` 自动补充）
  - `specs`（内含 `uom_unit_id`）

示例（节选）：

```json
{
  "id": 4,
  "name": "竹江-小笨熊抽纸（本色）",
  "uom_category_id": 3,
  "uom_base_unit_id": 4,
  "specs": [
    { "name": "1提装（10包）", "wholesale_price": 6, "uom_unit_id": 5 },
    { "name": "1件装（100包）", "wholesale_price": 52, "uom_unit_id": 4 },
    { "name": "3件装", "wholesale_price": 153, "uom_unit_id": 7 }
  ]
}
```

#### 4.2 价格展示策略（概述）

页面：`mini_app/pages/index/index.vue`、`category/category.vue`、`search/results.vue`

统一策略：

1. 取出 `baseUnitId = product.uom_base_unit_id || product.uomBaseUnitId`
2. 从 `specs` 中筛选出 `uom_unit_id == baseUnitId` 的规格作为候选集合（若为空则退回所有规格）
3. 根据当前用户类型：
   - 批发用户：优先取 `wholesale_price` 的最小值
   - 零售用户：优先取 `retail_price` 的最小值
   - 若都为空，再兜底用 `spec.price` 或商品自身 `price`
4. 结果写入 `product.displayPrice` 用于前端展示。

这样可以保证：**前台默认展示的是“基准单位规格”的最低价**。

### 5. 后台管理的防错设计

文件：`admin_console/src/views/Products.vue`

- 商品表单字段：
  - `productForm.uomCategoryId`：选择单位类别
  - `productForm.specs[n].uom_unit_id`：每个规格绑定的单位
- 为防止“先选好规格单位、再切换单位类别”导致脏数据，前端做了以下处理：
  - 监听 `productForm.uomCategoryId` 变化：
    - 计算当前类别下的合法单位 ID 集合；
    - 对 `specs` 遍历：
      - 如果某个 `spec.uom_unit_id` 不在当前类别单位列表中，则自动清空该字段；
      - 并提示“有 N 个规格的单位需要重新选择”。

> 目标：保证「商品的单位类别」与「规格中使用的单位」保持一致，避免出现 `uom_category_id` 指向 A 类别，而 `uom_unit_id` 仍然指向旧类别 B 的问题。

### 6. 后续扩展建议

- **支持更多单位类别**：
  - 例如“重量类（克/千克/吨）”“容量类（毫升/升/桶）”等，只需在 `uom_categories` / `uom_units` 配好，再在后台绑定到商品即可。
- **支持按基准单位换算展示**：
  - 当前价格展示只做“按基准单位规格选价”，后续可以考虑在前端展示类似「¥X / 包，约 ¥Y / 件」的换算价，使用 `ratio` 与 `delivery_count` 即可算出。
- **接口层安全校验**：
  - 若后续有需要，可以在创建/更新商品 API 中增加校验：
    - 所有 `spec.uom_unit_id` 必须属于当前 `uom_category_id`；
    - 否则直接拒绝请求，防止绕过前端校验。
